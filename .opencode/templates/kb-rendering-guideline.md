# KB Rendering Guideline

## 目的

这份规范用于约束写入知识库的 Markdown 内容格式，使其在当前前端渲染中更稳定、更自然、更易读。

## 默认语言

- 总结、快照、daily continuity、任务记录、决策记录、主题沉淀默认使用中文
- 只有用户明确要求或项目本身有固定语言约束时，才切换到其他语言

## 推荐结构

优先使用短标题 + 短段落 + 简洁列表。

推荐 section：

- 目标
- 核心结论
- 触发规则
- 实施内容
- 验证结果
- 关键文件
- 关联文档
- 下一步

不是每篇文档都必须包含全部 section，应按对象类型裁剪。

## 渲染友好规则

### 1. 少用反引号包裹工具名

不要在正文里反复写带代码样式的工具名，例如：

- sync_runtime_event
- sync_kb_object

因为当前渲染器会把它们显示成明显的代码样式，阅读体验较差。

优先改写为自然语言：

- 运行时事件同步
- 结构化对象同步
- 路径解析工具
- 对象状态查询工具

只有在以下场景才使用反引号：

- 文档标题中的稳定标识
- 文件路径
- 命令行
- 必须精确引用的 API / tool 名称

### 2. 避免一行里堆多个代码片段

不推荐：

运行时触发统一通过 sync_runtime_event 承接。

推荐：

- 运行时触发统一通过运行时事件同步承接。
- 触发类型包括 compression、checkpoint 和 manual。

### 3. 一条列表只表达一个意思

避免一条 bullet 里塞太多从句。

推荐把“原因 / 结果 / 例外”拆开写。

### 4. 控制列表长度

- 单个 section 中的列表优先控制在 3 到 6 条
- 如果超过 6 条，考虑拆成子 section

### 5. 少用大表格

当前渲染中的表格适合信息对照，不适合长段说明。

推荐：

- 需要对比时再使用表格
- 解释性内容优先用短列表或短段落

### 6. 避免嵌套层级过深

最多使用两层：

- 二级标题
- 简单列表

尽量避免：

- 三级列表
- 列表里再塞长段落

### 7. 标题尽量中文化

优先使用：

- 目标
- 核心结论
- 触发规则
- 实施内容
- 验证结果

除非该文档本身就是英文技术说明，否则不建议混用中英标题。

## 推荐对象写法

### Topic

- 用来讲“是什么”
- 重点写机制、边界、模型、长期规则

### Decision

- 用来讲“为什么这样定”
- 重点写原因、取舍、影响

### Task

- 用来讲“这次具体做了什么”
- 重点写实施内容、验证结果、状态

### Daily

- 用来讲“今天发生了什么、接下来做什么”
- 保持简短，不要堆长期机制解释

### Snapshot

- 用来讲“这个时点要保留什么上下文”
- 重点是恢复工作，不是完整复盘

## 推荐语气

- 简洁
- 中文自然句
- 少术语堆叠
- 少英文缩写连发
- 少像 API 文档，多像可读的知识卡片

## 一条判断标准

如果一段内容看起来像：

- 工具调用手册
- prompt 原文
- 接口字段表

那它通常不够适合作为最终知识库正文，需要再改写成面向阅读的表达。
