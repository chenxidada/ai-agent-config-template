# /review — 独立代码审查

当用户使用 `/review` 时，对当前代码变更进行独立审查，产出结构化审查报告。

## 适用场景
- 手动写完代码后想让 AI 审查一下
- 不在 `/feature` 流程中，但需要 reviewer 视角
- 对特定文件/模块进行 code review
- PR 前的自检

## 流程

### 第一步：确定审查范围

1. 如果用户指定了文件路径/模块 → 审查指定范围
2. 如果用户未指定 → 检查 `git diff` 获取当前未提交的改动作为审查范围
3. 如果无改动也无指定 → 询问用户要审查什么

### 第二步：执行审查

调用 @reviewer 进行代码审查。

审查维度（合并视角）：
- **实现正确性**：函数体有真实逻辑？边界条件覆盖？
- **设计一致性**：是否遵循项目现有模式和规范？
- **集成连通性**：数据路径是否完整可追踪？

### 第三步：输出报告

产出格式：

```markdown
# 代码审查报告

## 判决：PASS / MUST-FIX / SHOULD-FIX

## 审查范围
<文件列表或描述>

## Must-Fix 问题
（如有）

## Should-Fix 建议
（如有）

## 总结
<2-3 句概括>
```

### 第四步：向用户呈现

1. 读取审查报告
2. 展示判决 + 关键发现
3. 如有 MUST-FIX：建议修复方向

---

## 约束
- reviewer 是只读 agent，不修改代码
- 发现问题时只报告，不自动修复
- 判决标准与 `/feature` 流程中的 reviewer 一致
